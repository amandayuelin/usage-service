# SPEC.md

## 1. Problem Statement

Restate the problem in 3-5 clear sentences.

Explain:
- what data is ingested
- how it is processed
- what users can query
- what the service is responsible for

## 2. Goals

List the functional goals of the service.

Include goals such as:
- ingest data
- validate input
- persist raw records
- process records
- expose query endpoints
- handle duplicates or idempotency if applicable
- provide health/readiness endpoints
- support automated testing
- support deployment

## 3. Non-Goals

List what should not be implemented in the initial time-boxed version unless explicitly required.

Usually defer:
- Redis
- message queue
- worker service
- Kubernetes
- OAuth/RBAC
- Prometheus/Grafana
- OpenTelemetry
- complex CI/CD
- advanced admin UI
- multi-region deployment

State that these may be future production improvements.

## 4. Assumptions

State assumptions explicitly.

Include assumptions about:
- unique identifiers
- idempotency key
- storage choice
- synchronous vs asynchronous processing
- default limits
- deployment target
- pagination
- validation boundaries

## 5. API Contract

Define the REST API before implementation.

Include endpoint categories when applicable:

- GET /health
- GET /ready
- POST ingestion endpoint
- GET resource by id endpoint
- GET list endpoint with pagination
- GET result/status/summary endpoint based on the problem

For each endpoint, define:
- method and path
- purpose
- request body if applicable
- success response example
- error cases
- HTTP status codes

Use versioned API paths such as /api/v1 where appropriate.

## 6. Data Model

Define the core entities.

For each entity, include:
- fields
- types at a high level
- required fields
- relationships
- indexes
- uniqueness constraints

If the service ingests external events or records, define a natural idempotency key such as:
- source + external_id
- account_id + external_id
- client_id + request_id

## 7. Processing Rules

Define how incoming data is processed.

Include:
- supported event/resource/data types if applicable
- transformations
- aggregations
- status calculation
- result generation
- edge cases
- failure behavior

Keep processing synchronous in the initial version unless the problem explicitly requires async processing.

## 8. Idempotency / Duplicate Handling

Define:
- idempotency key
- first request behavior
- duplicate request behavior
- expected status codes
- how duplicate processing is avoided

Prefer idempotent replay over 409 Conflict for ingestion APIs unless the problem explicitly requires conflict behavior.

## 9. Validation Rules

Define:
- required fields
- type checks
- non-empty strings
- supported enum values
- positive numeric values
- valid timestamps
- payload shape
- pagination limits
- domain-specific validation

## 10. Error Handling

Use one consistent JSON error shape:

{
  "error": {
    "code": "string",
    "message": "string",
    "request_id": "string",
    "details": {}
  }
}

Define common error cases and status codes:
- 400 validation_error
- 404 not_found
- 409 conflict only if duplicate behavior is not idempotent
- 500 internal_server_error
- 503 dependency_unavailable

Errors should be safe for clients and should not expose stack traces.

## 11. Operational Requirements

Define live-service behavior:

- request_id middleware
- structured request logging
- /health endpoint
- /ready endpoint with dependency checks
- environment-based configuration
- safe error responses
- stateless API design
- pagination
- bounded inputs where appropriate

Suggested environment variables:
- APP_ENV
- PORT
- DATABASE_URL
- LOG_LEVEL
- MAX_PAGE_SIZE

## 12. Testing Strategy

Define unit tests and API tests.

Prioritize tests for:
- health endpoint
- readiness endpoint
- successful ingestion
- validation failure
- duplicate/idempotent replay behavior if applicable
- get by id success
- get by id not found
- get result/status/summary
- list endpoint with pagination
- processor/business rules

## 13. Automation and Workflow

Define expected project workflow:

- Dockerfile
- docker-compose.yml
- Makefile
- README.md
- one-command local run
- one-command test
- Docker build/run commands
- smoke test commands

Suggested Makefile commands:
- run
- test
- docker-build
- docker-run
- compose-up
- smoke-local

## 14. Deployment Plan

Target DigitalOcean App Platform unless the problem specifies otherwise.

Prefer:
- Dockerized deployment
- environment variables for configuration
- PostgreSQL if feasible
- SQLite fallback only if PostgreSQL setup becomes a deployment risk

Include smoke tests for deployed service:
- GET /health
- GET /ready
- POST ingestion endpoint
- GET query endpoint

## 15. Scalability Considerations

Explain how the initial version scales:

- stateless API
- external durable database
- pagination
- indexed queries
- bounded request payloads
- horizontal scaling on managed app platform

Also explain future production evolution:

- queue + worker
- retry policy
- dead-letter handling
- Redis/Valkey for cache or rate limiting
- metrics and tracing
- authentication
- database migrations
- CI/CD pipeline

## 16. Trade-offs

Explain important trade-offs.

Include:
- why processing is synchronous in the initial version
- why queue/worker is deferred unless required
- why the design focuses on a production-candidate slice rather than a half-finished distributed system
- what would be added next in a real production environment

## 17. Future Production Improvements

List future improvements separately from initial implementation.

Include only if relevant:
- async queue-backed processing
- worker service
- retries
- dead-letter queue
- rate limiting
- caching
- authentication/authorization
- metrics
- tracing
- schema migrations
- CI/CD
- stronger deployment strategy
