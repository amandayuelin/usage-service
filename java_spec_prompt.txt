## 8. Processing Rules

## 9. Idempotency / Duplicate Handling

Define duplicate behavior if the problem includes retries, external events, repeated submissions, or client-generated IDs.

Include:
- idempotency key
- first request behavior
- duplicate request behavior
- expected status codes

- how database uniqueness constraints support correctness

Choose one MVP behavior:
- return 409 conflict for duplicate submissions, or
- return 200 with existing result and idempotentReplay=true

Prefer whichever is simpler and easier to explain, but state the trade-off.

If idempotency is not applicable, explicitly say so.

## 10. Transaction and Concurrency Considerations

Define the transaction boundary.

Include:
- which operations must happen atomically
- which records should be inserted or updated together
- what can go wrong under concurrent requests
- what the MVP handles
- what production should improve

For the MVP:
- use a reasonable @Transactional boundary in the service layer
- do not claim full concurrency safety unless row-level locking, optimistic locking, or atomic update is implemented and tested

If the problem involves quota, inventory allocation, scheduling, reservations, resource limits, or other correctness-critical decisions, explain stronger production concurrency options:
- row-level locking
- optimistic locking with @Version
- atomic update pattern
- uniqueness constraints
- serializable transactions, if justified
